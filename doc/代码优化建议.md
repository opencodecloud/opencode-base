## 一、HIGH - 高优先级优化

### 1.1 [validation] ConstraintValidatorFactory 反射实例化性能瓶颈

**文件**: `opencode-base-validation/.../spi/DefaultConstraintValidatorFactory.java`

**问题**: 每次约束校验都通过反射创建新的 Validator 实例，包括 `getDeclaredConstructor()` + `newInstance()`。批量校验 1000 个 Bean（每个 10 个字段）= 10,000 次反射调用。

```java
public <T extends ConstraintValidator<?, ?>> T getInstance(Class<T> key) {
    return (T) createInstance(key);  // 每次反射创建新实例
}
```

**建议**: 至少缓存 Constructor 对象，最好缓存无状态 Validator 单例：

```java
private static final ConcurrentHashMap<Class<?>, Constructor<?>> CONSTRUCTORS = new ConcurrentHashMap<>();

private ConstraintValidator<?, ?> createInstance(Class<?> key) {
    Constructor<?> ctor = CONSTRUCTORS.computeIfAbsent(key, cls -> {
        try { return cls.getDeclaredConstructor(); }
        catch (NoSuchMethodException e) { throw new ValidationException("...", e); }
    });
    return (ConstraintValidator<?, ?>) ctor.newInstance();
}
```

**收益**: 批量校验场景 10-100x 性能提升。

### 1.2 [parallel] RateLimitedExecutor 锁竞争严重

**文件**: `opencode-base-parallel/.../executor/RateLimitedExecutor.java`

**问题**: 每次 `acquire()` 都需要获取锁，即使令牌充足。高并发下所有提交线程串行化。

**建议**: 增加无锁快速路径，仅在需要补充令牌时加锁：

```java
double available = storedPermits;  // volatile read
if (available >= 1.0) {
    lock.lock();
    try {
        refillTokens();
        if (storedPermits >= 1.0) {
            storedPermits -= 1.0;
            return true;
        }
    } finally {
        lock.unlock();
    }
}
```

**收益**: 高并发场景 40-60% 吞吐量提升。

### 1.3 [crypto] SecureRandoms.nextBytes() 热路径不必要的实例分配

**文件**: `opencode-base-crypto/.../random/SecureRandoms.java:161-168`

**问题**: 每次调用 `nextBytes()` 都通过 `getDefault()` 创建新的 SecureRandom 实例。在 IV/nonce 生成等热路径上产生大量分配。

```java
public static byte[] nextBytes(int numBytes) {
    byte[] bytes = new byte[numBytes];
    getDefault().nextBytes(bytes);  // 每次创建新 SecureRandom
    return bytes;
}
```

**建议**: 使用 ThreadLocal 缓存 SecureRandom 实例（参考 `AesGcmCipher:41` 的做法）。

**收益**: 加密操作 50-80% 分配开销降低。

### 1.4 [core] ConverterRegistry 缺少类型层次查找

**文件**: `opencode-base-core/.../convert/ConverterRegistry.java:73-86`

**问题**: 仅做精确类型匹配。注册了 `Number` 的转换器，查询 `Integer` 时返回 null。

**建议**: 实现类型继承链查找：

```java
public static <T> Converter<T> getConverter(Type type) {
    Converter<T> converter = (Converter<T>) CONVERTERS.get(type);
    if (converter != null) return converter;
    if (type instanceof Class<?> cls) {
        for (Class<?> supertype : cls.getInterfaces()) {
            converter = (Converter<T>) CONVERTERS.get(supertype);
            if (converter != null) return converter;
        }
        Class<?> superclass = cls.getSuperclass();
        while (superclass != null && superclass != Object.class) {
            converter = (Converter<T>) CONVERTERS.get(superclass);
            if (converter != null) return converter;
            superclass = superclass.getSuperclass();
        }
    }
    return null;
}
```

**收益**: 多态转换器正常工作，消除重复注册。

### 1.5 [cache] DefaultCache 缓存访问路径双重查找

**文件**: `opencode-base-cache/.../internal/DefaultCache.java:763-777`

**问题**: `recordAccess()` 对 store 做二次 get，且 `get()` 调用方已持有 holder 引用却未传入。同时创建了不必要的 CacheEntry 对象。

```java
private void recordAccess(K key, V value) {
    ValueHolder<V> holder = store.get(key);  // 第一次查找（调用方已查过）
    if (holder != null) {
        holder.lastAccessTime = System.currentTimeMillis();
        holder.accessCount.incrementAndGet();
    }
    CacheEntry<K, V> entry = new CacheEntry<>(key, value);  // 不必要的对象创建
    evictionPolicy.recordAccess(entry);
    if (config.expireAfterAccess() != null) {
        updateExpiration(key);  // 可能第三次查找
    }
}
```

**建议**: 将 holder 引用从 `get()` 传递到 `recordAccess()`，避免重复查找和对象创建。

**收益**: 每次缓存访问减少 1-2 次 HashMap 查找。

### 1.6 [net] RateLimitedClient 信号量泄漏

**文件**: `opencode-base-net/.../pool/RateLimitedClient.java:165-172`

**问题**: `send()` 方法获取信号量后，若请求失败不释放许可。随时间推移许可逐渐耗尽，最终阻塞所有请求。

```java
public <T> HttpResponse<T> send(HttpRequest request, HttpResponse.BodyHandler<T> bodyHandler) {
    acquire();
    try {
        return httpClient.send(request, bodyHandler);
    } catch (Exception e) {
        throw new OpenHttpException("Request failed", e);  // 许可泄漏！
    }
}
```

**建议**: 使用 finally 块保证释放：

```java
public <T> HttpResponse<T> send(...) {
    acquire();
    try {
        return httpClient.send(request, bodyHandler);
    } finally {
        rateLimiter.release();
    }
}
```

**收益**: 消除渐进式服务降级风险。

### 1.7 [net] MultipartBody 大文件全部加载到内存

**文件**: `opencode-base-net/.../body/MultipartBody.java:128-170`

**问题**: 构造函数调用 `buildBody()` 将所有文件内容读入 byte[]，500MB 文件直接 OOM。

```java
byte[] content = Files.readAllBytes(file);  // 全部读入内存
```

**建议**: 实现流式 BodyPublisher，按需读取各 part 数据而非全部缓冲。

**收益**: 支持大文件上传，消除 OOM 风险。

### 1.8 [io] ChunkedFileProcessor 虚拟线程创建开销

**文件**: `opencode-base-io/.../file/ChunkedFileProcessor.java:472-521`

**问题**: 对每个 chunk 创建独立虚拟线程。百万小 chunk 场景下虚拟线程创建开销巨大。虽有 Semaphore 限制并行度，但线程创建是提前全量进行的。

**建议**: 使用有界线程池或任务批处理方式，避免全量 Future 预创建。

**收益**: 大文件处理性能显著提升。

---

### 1.9 [geo] GeoHashEncoder decode/getBoundingBox 代码完全重复

**文件**: `opencode-base-geo/.../geohash/GeoHashEncoder.java:86-124, 172-209`

**问题**: `decode()` 和 `getBoundingBox()` 包含完全相同的二进制搜索逻辑（各约 30 行），仅返回值不同。

**建议**: 提取私有方法 `decodeRanges(String geoHash)` 返回 record，两个公开方法共用。

**收益**: 消除 50% 代码重复，降低维护成本。

---

### 1.10 [expression] 表达式缓存驱逐策略非 LRU

**文件**: `opencode-base-expression/.../OpenExpression.java:331-348`

**问题**: 缓存满时删除前 50% 的 key，但没有访问顺序追踪，实际是随机删除，不是 LRU。

```java
if (EXPRESSION_CACHE.size() >= MAX_CACHE_SIZE) {
    EXPRESSION_CACHE.keySet().stream()
            .limit(MAX_CACHE_SIZE / 2)
            .toList()
            .forEach(EXPRESSION_CACHE::remove);  // 非 LRU，随机删除
}
```

**建议**: 使用 `LinkedHashMap(capacity, 0.75f, true)` 实现访问顺序 LRU，或使用专用 LRU 缓存。

**收益**: 持续负载下缓存命中率显著提升，避免缓存抖动。

### 1.11 [reflect] BeanCopier.convertIfNeeded() 类型检查无缓存

**文件**: `opencode-base-reflect/.../bean/BeanCopier.java:169-216`

**问题**: 14+ 个顺序类型检查，每次属性复制都重复执行。批量复制时开销累积。

**建议**: 使用 `ConcurrentHashMap<Class<?>, Function<Object, Object>>` 缓存类型转换函数。

**收益**: 批量 BeanCopy 场景性能显著提升。

## 二、MEDIUM - 中优先级优化

### 2.1 [cache] WTinyLfuEvictionPolicy 使用非线程安全 LinkedHashSet

**文件**: `opencode-base-cache/.../eviction/WTinyLfuEvictionPolicy.java:59-63`

**问题**: `window`、`probation`、`protected_` 三个 LinkedHashSet 非线程安全，但 DefaultCache 本身是线程安全的。文档标注"外部同步"，但调用方未明确加锁。

**建议**: 在驱逐策略方法中增加同步，或使用线程安全集合。

### 2.2 [observability] MetricsRegistry key 构造开销

**文件**: `opencode-base-observability/.../metrics/MetricsRegistry.java:188-197`

**问题**: 每次指标访问都创建 StringBuilder 并排序 tags，含流操作和排序。

**建议**: 预排序 tags 或使用哈希函数替代字符串拼接。

---

### 2.3 [observability] HistogramMetric 线性桶查找

**文件**: `opencode-base-observability/.../metrics/HistogramMetric.java:61-74`

**问题**: O(n) 线性搜索定位桶，默认 11 个桶平均 6 次比较。

**建议**: 使用 `Arrays.binarySearch()` 降至 O(log n) = 3-4 次比较。

### 2.4 [observability] TraceContext 双路径 ScopedValue/ThreadLocal

**文件**: `opencode-base-observability/.../tracing/TraceContext.java:20-67`

**问题**: 同时维护 ScopedValue 和 ThreadLocal，静默回退到 ThreadLocal 时可能隐藏问题。

**建议**: 完全迁移到 ScopedValue（Java 25），移除 ThreadLocal 回退路径。

---

### 2.5 [db] JdbcTemplate 重复 EntityMeta 查找

**文件**: `opencode-base-db/.../core/JdbcTemplate.java:334-434`

**问题**: `findById()`、`save()`、`updateEntity()`、`delete()` 等方法各自调用 `EntityMeta.of()`。批量操作时重复解析元数据。

**建议**: 增加接受预解析 EntityMeta 的重载方法。

### 2.6 [db] SimpleConnectionPool 驱逐期间连接临时不可用

**文件**: `opencode-base-db/.../pool/SimpleConnectionPool.java:355-375`

**问题**: `evictIdleConnections()` 使用 poll-offer 模式，循环期间连接对其他线程临时不可用。

**建议**: 使用 peek + 单独移除策略替代 poll-offer 模式。

### 2.7 [config] CollectionConverters 返回可变集合

**文件**: `opencode-base-config/.../converter/impl/CollectionConverters.java:37-54`

**问题**: 使用 `Collectors.toList()` 返回可变 ArrayList，配置值应为不可变。

**建议**: 使用 `Collectors.toUnmodifiableList()`。

### 2.8 [io] FastByteArrayOutputStream 缓冲区共享

**文件**: `opencode-base-io/.../stream/FastByteArrayOutputStream.java:139-144`

**问题**: `toByteArray()` 在 count == buffer.length 时直接返回内部缓冲区引用，调用方可修改内部状态。

```java
public byte[] toByteArray() {
    if (count == buffer.length) return buffer;  // 返回内部引用
    return Arrays.copyOf(buffer, count);
}
```

**建议**: 始终返回副本，或提供明确命名的 `getInternalBuffer()` 用于高级用途。

**文件**: `opencode-base-net/.../HttpClient.java:500-587`

**问题**: `connectTimeout()`、`readTimeout()` 等方法每次都重建完整 HttpConfig 对象。

### 2.10 [email] OpenEmail 同步策略混乱

**问题**: 字段使用 volatile，方法使用 synchronized，但部分读取路径（如 `getSender()`）未同步。

### 2.11 [string] AhoCorasick 重叠匹配合并逻辑有误

### 2.12 [string] StringTemplate 变量提取无缓存

private Set<String> cachedVariables;
public Set<String> getVariables() {

    return cachedVariables;
