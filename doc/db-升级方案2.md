# DB 组件升级方案：新一代 ORM 框架

## 1. 设计哲学

**我们不是 MyBatis 的复刻品，而是它的继承者。**

MyBatis 诞生于 Java 6 时代，MyBatis-Plus 诞生于 Java 8 时代。它们受限于历史包袱：XML 重度依赖、反射开销大、API 冗余、不支持 Record、不支持模式匹配、不支持虚拟线程。

`opencode-base-db` 面向 Java 25+，站在巨人肩膀上重新设计：

| 理念 | MyBatis/Plus | opencode-base-db |
|------|-------------|------------------|
| 配置方式 | XML 优先，注解辅助 | **代码优先，XML 可选**（兼容迁移） |
| 类型安全 | 字符串列名，运行时报错 | **Lambda 列引用，编译期检查** |
| 映射模型 | 仅 JavaBean | **Bean + Record + Map 三位一体** |
| 动态 SQL | OGNL + XML 标签 | **Java 代码 + 表达式引擎**（兼容 XML 语法） |
| 并发模型 | 传统线程池 | **虚拟线程原生支持** |
| API 风格 | 接口多、类多、概念多 | **最小 API，最大能力** |
| 模块化 | 单体 JAR | **JPMS 模块，按需引入** |

**兼容目标**：支持 MyBatis mapper.xml 语法，支持 MyBatis-Plus 核心编程模式（BaseMapper、Wrapper、Lambda），但包名和 API 是我们自己的，不是照搬。用户用我们的方式写新代码，用兼容层迁移旧代码。

## 2. 模块架构

```
opencode-base-db              (现有模块，增强)
  ├── 核心引擎 (JdbcTemplate, EntityMeta, SQL Builder, 事务, 连接池, 方言)
  ├── 【新增】BaseMapper<T> + CRUD 自动生成
  ├── 【新增】Wrapper 条件构造器 (Query/Update/Lambda)
  ├── 【新增】分页 (Page<T>)
  ├── 【新增】Plus 注解 (@TableName, @TableId, @TableField, @TableLogic, @Version)
  ├── 【新增】自动填充 (AutoFillHandler)
  ├── 【新增】逻辑删除
  ├── 【新增】乐观锁
  ├── 【新增】拦截器链 (Interceptor/InnerInterceptor)
  └── 【新增】IService<T> 服务层模板

opencode-base-db-mapper       (新建模块)
  ├── Mapper XML 解析 (<select>/<insert>/<update>/<delete>)
  ├── 动态 SQL 节点 (<if>/<choose>/<where>/<set>/<foreach>/<trim>/<bind>)
  ├── Mapper 接口代理 (MapperProxy)
  ├── Mapper 注解 (@Select/@Insert/@Update/@Delete)
  ├── TypeHandler 类型转换体系
  ├── ResultMap 结果集映射
  └── SQL Session 兼容层
```

### 为何这样划分

| 决策 | 原因 |
|------|------|
| BaseMapper/Wrapper 放 `opencode-base-db` | 纯 Java 逻辑，仅依赖现有 SQL Builder + EntityMeta，无需额外依赖 |
| XML Mapper 独立为 `opencode-base-db-mapper` | 需要 `opencode-base-xml`（DOM 解析）+ `opencode-base-expression`（动态表达式）+ `opencode-base-reflect`（代理），按需引入 |
| 不建新的 `mybatis`/`mybatisplus` 包名模块 | 我们是自己的框架，不是换皮 MyBatis |

### 依赖关系

```
opencode-base-db-mapper
    ├── opencode-base-db          (核心引擎)
    ├── opencode-base-xml         (XML mapper 解析)
    ├── opencode-base-expression  (动态 SQL 表达式求值，替代 OGNL)
    ├── opencode-base-reflect     (Mapper 代理 + Lambda 列名提取)
    └── opencode-base-cache       (二级缓存)
```

用户只用 BaseMapper + Wrapper：
```xml
<dependency>
    <groupId>cloud.opencode.base</groupId>
    <artifactId>opencode-base-db</artifactId>
</dependency>
```

用户需要 XML Mapper 兼容：
```xml
<dependency>
    <groupId>cloud.opencode.base</groupId>
    <artifactId>opencode-base-db-mapper</artifactId>
</dependency>
```

### 新增 API 与现有 API 的关系

`opencode-base-db` 已有一套注解和监听器机制。新增的 API 与现有 API 的关系如下：

#### 注解关系（共存，不替代）

| 现有注解 | 新增注解 | 关系 |
|---------|---------|------|
| `@Table(value, schema, catalog)` | `@TableName(value, schema)` | **共存**。`@Table` 继续用于现有 SQL Builder 场景；`@TableName` 用于 BaseMapper/Wrapper 场景。`EntityMeta` 同时识别两者，优先级：`@TableName` > `@Table` |
| `@Column(value, insertable, updatable, nullable)` | `@TableField(value, exist, fill, update, select)` | **共存**。`@Column` 保留用于 SQL Builder 和 RecordMapper；`@TableField` 增加了自动填充、查询排除等 Plus 特性。`EntityMeta` 同时识别两者 |
| `@Id(autoGenerated, column)` | `@TableId(value, type)` | **共存**。`@Id` 用于现有场景；`@TableId` 增加了雪花/UUID 等 ID 策略。`EntityMeta` 同时识别两者 |
| `@Transient` | `@TableField(exist = false)` | **等价**。两种写法都表示字段不参与数据库映射 |

> **建议**：新代码统一使用 `@TableName`/`@TableId`/`@TableField`，旧代码的 `@Table`/`@Column`/`@Id` 无需修改，继续有效。

#### 监听器 vs 拦截器

| 现有 API | 新增 API | 关系 |
|---------|---------|------|
| `SqlExecutionListener`（beforeExecute/afterExecute/onError） | `DbInterceptor`（beforeExecute/afterExecute） | **共存，不同层级**。`SqlExecutionListener` 是**只读观察者**（用于日志、监控），不能修改 SQL；`DbInterceptor` 是**可修改拦截器**（可改写 SQL 和参数）。执行顺序：Interceptor 链 → Listener 通知 |
| `AuditLogListener` | `SlowSqlInterceptor` | **共存**。`AuditLogListener` 记录 SQL 执行日志；`SlowSqlInterceptor` 专注慢 SQL 告警。两者职责不同 |

#### 脱敏机制

| 现有 API | 新增 API | 关系 |
|---------|---------|------|
| `@Sensitive` + `SensitiveType` + `SensitiveDataMapper` | `SensitiveMaskInterceptor` | **共存互补**。`@Sensitive` + `SensitiveDataMapper` 是手动脱敏（需在 RowMapper 中显式调用）；`SensitiveMaskInterceptor` 是**全局自动脱敏**（拦截器自动处理所有 `@Sensitive` 字段）。推荐新代码使用拦截器方式 |

### 3.1 新增注解（`cloud.opencode.base.db.mapper.annotation`）

```java
// 表名映射（增强现有 @Table）
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface TableName {
    String value() default "";        // 表名，空则按命名策略推导
    String schema() default "";
}

// 主键策略（增强现有 @Id）
@Target({ElementType.FIELD, ElementType.RECORD_COMPONENT})
@Retention(RetentionPolicy.RUNTIME)
public @interface TableId {
    String value() default "";        // 列名
    IdType type() default IdType.AUTO;
}

public enum IdType {
    AUTO,           // 数据库自增
    NONE,           // 无策略（用户手动赋值）
    ASSIGN_ID,      // 雪花算法
    ASSIGN_UUID     // UUID
}

// 字段映射（增强现有 @Column）
@Target({ElementType.FIELD, ElementType.RECORD_COMPONENT})
@Retention(RetentionPolicy.RUNTIME)
public @interface TableField {
    String value() default "";        // 列名
    boolean exist() default true;     // false = @Transient
    FieldFill fill() default FieldFill.DEFAULT;
    String update() default "";       // update 时的 SQL 片段，如 "NOW()"
    boolean select() default true;    // 是否参与查询
}

public enum FieldFill {
    DEFAULT, INSERT, UPDATE, INSERT_UPDATE
}

// 逻辑删除
@Target({ElementType.FIELD, ElementType.RECORD_COMPONENT})
@Retention(RetentionPolicy.RUNTIME)
public @interface TableLogic {
    String value() default "0";       // 未删除值
    String delval() default "1";      // 已删除值
}

// 乐观锁
@Target({ElementType.FIELD, ElementType.RECORD_COMPONENT})
@Retention(RetentionPolicy.RUNTIME)
public @interface Version { }

// 排序
@Target({ElementType.FIELD, ElementType.RECORD_COMPONENT})
@Retention(RetentionPolicy.RUNTIME)
public @interface OrderBy {
    boolean asc() default true;
    int sort() default Integer.MAX_VALUE;
}

// 枚举值映射 — 指定枚举中哪个字段作为数据库存储值
@Target({ElementType.FIELD, ElementType.RECORD_COMPONENT})
@Retention(RetentionPolicy.RUNTIME)
public @interface EnumValue { }

// 示例：
// public enum Status {
//     ACTIVE(1, "Active"), INACTIVE(0, "Inactive");
//     @EnumValue private final int code;
//     private final String desc;
// }
// → 数据库存储 code 值（1 或 0），而非 name() 或 ordinal()
```

### 3.2 BaseMapper<T>（`cloud.opencode.base.db.mapper`）

```java
/**
 * 通用 Mapper 接口，提供基础 CRUD 操作。
 * <p>Generic Mapper interface providing basic CRUD operations.</p>
 *
 * <pre>{@code
 * public interface UserMapper extends BaseMapper<User> { }
 *
 * // 使用
 * UserMapper mapper = db.mapper(UserMapper.class);
 * mapper.insert(user);
 * mapper.selectById(1L);
 * mapper.selectList(LambdaQuery.of(User.class).eq(User::getAge, 18));
 * }</pre>
 */
public interface BaseMapper<T> {

    // ===== 插入 =====
    int insert(T entity);
    int insertOrUpdate(T entity);                                    // Upsert：不存在则插入，存在则更新

    // ===== 删除 =====
    int deleteById(Serializable id);
    int deleteBatchIds(Collection<? extends Serializable> ids);
    int delete(Wrapper<T> wrapper);
    int deleteByMap(Map<String, Object> columnMap);                  // 按 Map 条件删除

    // ===== 更新 =====
    int updateById(T entity);
    int update(T entity, Wrapper<T> wrapper);
    int update(Wrapper<T> wrapper);                                  // 纯 Wrapper 更新（无需实体，用 set() 设值）

    // ===== 查询 =====
    T selectById(Serializable id);
    List<T> selectBatchIds(Collection<? extends Serializable> ids);
    List<T> selectByMap(Map<String, Object> columnMap);              // 按 Map 条件查询
    T selectOne(Wrapper<T> wrapper);
    boolean exists(Wrapper<T> wrapper);
    long selectCount(Wrapper<T> wrapper);
    List<T> selectList(Wrapper<T> wrapper);
    List<Object> selectObjs(Wrapper<T> wrapper);                     // 只取第一列
    List<Map<String, Object>> selectMaps(Wrapper<T> wrapper);
    Page<T> selectPage(Page<T> page, Wrapper<T> wrapper);
    Page<Map<String, Object>> selectMapsPage(Page<T> page, Wrapper<T> wrapper);  // 分页返回 Map
}
```

**实现原理**：`BaseMapper` 默认实现类内部持有 `JdbcTemplate` + `EntityMeta`，每个方法通过 EntityMeta 获取表名/列名，使用现有 SQL Builder（`Select`/`Insert`/`Update`/`Delete`）+ `Condition` 生成 SQL，交给 `JdbcTemplate` 执行。

#### SFunction —— Lambda 列引用

```java
/**
 * 可序列化的 Function，用于从 Lambda 方法引用中提取属性名。
 * <p>Serializable function for extracting property names from method references.</p>
 *
 * <pre>{@code
 * SFunction<User, String> fn = User::getName;
 * String propertyName = SFunction.propertyName(fn);  // "name"
 * }</pre>
 */
@FunctionalInterface
public interface SFunction<T, R> extends Function<T, R>, Serializable {
    static <T, R> String propertyName(SFunction<T, R> fn);
    static <T, R> String columnName(SFunction<T, R> fn, Class<T> entityClass);
}
```

**实现**：内部使用 `opencode-base-reflect` 的 `SerializedLambdaWrapper` 从 `User::getName` 提取出属性名 `"name"`，再通过 `EntityMeta` 的命名策略转为列名 `"user_name"`。

### 3.4 分页（`cloud.opencode.base.db.page`）

```java
/**
 * 分页对象，同时作为分页参数和分页结果。
 * <p>Page object serving as both pagination parameters and results.</p>
 *
 * <pre>{@code
 * Page<User> page = Page.of(1, 10);  // 第1页，每页10条
 * page = mapper.selectPage(page, LambdaQuery.of(User.class).eq(User::getStatus, "active"));
 *
 * List<User> records = page.getRecords();
 * long total = page.getTotal();
 * long pages = page.getPages();
 * }</pre>
 */
public class Page<T> {
    public static <T> Page<T> of(long current, long size);
    public static <T> Page<T> of(long current, long size, boolean searchCount);

    public long getCurrent();         // 当前页码（从1开始）
    public long getSize();            // 每页条数
    public long getTotal();           // 总记录数
    public long getPages();           // 总页数
    public List<T> getRecords();      // 当前页数据
    public boolean hasNext();
    public boolean hasPrevious();

    // 内部使用
    public Page<T> setRecords(List<T> records);
    public Page<T> setTotal(long total);
}
```

**实现**：`selectPage` 内部先执行 `SELECT COUNT(*)`，再使用 `Dialect.getLimitSql()` 生成分页 SQL 执行查询。

### 3.5 自动填充（`cloud.opencode.base.db.fill`）

```java
/**
 * 自动填充处理器，在 insert/update 时自动设置字段值。
 * <p>Auto-fill handler for automatically setting field values on insert/update.</p>
 *
 * <pre>{@code
 * AutoFillHandler handler = AutoFillHandler.builder()
 *     .onInsert("createTime", LocalDateTime::now)
 *     .onInsert("createBy", () -> SecurityContext.getCurrentUser())

 *     .build();

    void fillUpdate(Object entity, EntityMeta meta);

    static Builder builder();

    interface Builder {
        Builder onInsert(String fieldName, Supplier<?> valueSupplier);
        Builder onUpdate(String fieldName, Supplier<?> valueSupplier);
        Builder onInsertUpdate(String fieldName, Supplier<?> valueSupplier);
        AutoFillHandler build();
    }
}
```

### 3.6 拦截器（`cloud.opencode.base.db.interceptor`）

```java
/**
 * SQL 拦截器，在 SQL 执行前后进行拦截处理。
 * <p>SQL interceptor for pre/post processing of SQL execution.</p>
 */
public interface DbInterceptor {
    /** 在 SQL 执行前调用，可修改 SQL 和参数 */
    default SqlInfo beforeExecute(SqlInfo sqlInfo) { return sqlInfo; }
    /** 在 SQL 执行后调用，可修改结果 */
    default Object afterExecute(SqlInfo sqlInfo, Object result) { return result; }
    /** 拦截器顺序，值越小越先执行 */
    default int order() { return 0; }
}

/** SQL 信息，可被拦截器修改 */
public record SqlInfo(String sql, Object[] params, SqlType type, Class<?> entityType) {
    public enum SqlType { SELECT, INSERT, UPDATE, DELETE }
    public SqlInfo withSql(String newSql);
    public SqlInfo withParams(Object[] newParams);
}
```

#### `@InterceptorIgnore` — 方法级拦截器跳过

```java
/**
 * 标记 Mapper 接口方法或类，跳过指定拦截器。
 * <p>Skip specific interceptors at method or class level.</p>
 *
 * <pre>{@code
 * public interface UserMapper extends BaseMapper<User> {
 *     // 管理员查全量数据：跳过租户和数据权限过滤
 *     @InterceptorIgnore(tenant = true, dataPermission = true)
 *     List<User> selectAllTenants();
 *
 *     // 物理删除：跳过逻辑删除拦截
 *     @InterceptorIgnore(logicDelete = true)
 *     int physicalDeleteById(@Param("id") Long id);
 *
 *     // 特殊报表查询：跳过查询行数限制
 *     @InterceptorIgnore(queryLimit = true)
 *     List<ReportRow> exportFullReport();
 * }
 * }</pre>
 */
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
public @interface InterceptorIgnore {
    boolean tenant() default false;            // 跳过 TenantInterceptor
    boolean dataPermission() default false;    // 跳过 DataPermissionInterceptor
    boolean logicDelete() default false;       // 跳过 LogicDeleteInterceptor
    boolean blockAttack() default false;       // 跳过 BlockAttackInterceptor
    boolean pagination() default false;        // 跳过 PaginationInterceptor
    boolean optimisticLock() default false;    // 跳过 OptimisticLockInterceptor
    boolean encrypt() default false;           // 跳过 EncryptInterceptor
    boolean sensitiveMask() default false;     // 跳过 SensitiveMaskInterceptor
    boolean illegalSql() default false;        // 跳过 IllegalSqlInterceptor
    boolean slowSql() default false;           // 跳过 SlowSqlInterceptor
    boolean queryLimit() default false;        // 跳过 QueryLimitInterceptor
    boolean dynamicTable() default false;      // 跳过 DynamicTableInterceptor
    String[] others() default {};              // 按类名跳过自定义拦截器
}
```

**实现**：`MapperProxy` 在方法调用前检查 `@InterceptorIgnore` 注解，将忽略标记写入 `SqlInfo` 上下文，各拦截器在 `beforeExecute` 中检查对应标记并决定是否跳过。

### 4.8 ResultMap — 完整结果集映射（`cloud.opencode.base.db.mapper.result`）

```java
/**
 * 结果集映射定义 — 完整支持 MyBatis <resultMap> 全部特性。
 *
 * <p>支持的 XML 子元素：</p>
 * <ul>
 *   <li>{@code <id column="..." property="...">} — 主键映射</li>
 *   <li>{@code <result column="..." property="...">} — 普通字段映射</li>
 *   <li>{@code <constructor>} / {@code <idArg>} / {@code <arg>} — 构造函数映射</li>
 *   <li>{@code <association>} — 一对一关联（嵌套查询 / 嵌套结果 / 内联 resultMap）</li>
 *   <li>{@code <collection>} — 一对多关联（嵌套查询 / 嵌套结果 / 内联 resultMap）</li>
 *   <li>{@code <discriminator>} / {@code <case>} — 鉴别器（多态映射）</li>
 * </ul>
 *
 * <p>支持的属性：</p>
 * <ul>
 *   <li>{@code id} — 唯一标识</li>
 *   <li>{@code type} — 目标 Java 类型</li>
 *   <li>{@code extends} — 继承另一个 resultMap</li>
 *   <li>{@code autoMapping} — 是否自动映射未声明的列</li>
 * </ul>
 */
public record ResultMapDef(
    String id,
    Class<?> type,
    String extendsId,                              // 继承的 resultMap id
    Boolean autoMapping,                           // 自动映射（null = 使用全局设置）
    // 构造函数映射
    ConstructorDef constructor,                    // <constructor>（可为 null）
    // 字段映射
    List<ResultMapping> idMappings,                // <id> 列表
    List<ResultMapping> resultMappings,            // <result> 列表
    // 关联映射
    List<AssociationDef> associations,             // <association> 列表
    List<CollectionDef> collections,               // <collection> 列表
    // 鉴别器
    DiscriminatorDef discriminator                 // <discriminator>（可为 null）
) { }

public record ResultMapping(

    String property,

    JdbcType jdbcType,                 // 可为 null

    boolean isId                       // 是否为 <id>
) { }

/**
 * 构造函数映射 — {@code <constructor>}
 *
 * <pre>{@code
 * <resultMap id="userMap" type="User">

 *     <idArg column="id" javaType="long"/>

    ) { }
}

/**
 * 一对一关联 — {@code <association>}
 *
 * <p>三种使用方式：</p>
 * <ol>
 *   <li>嵌套查询：{@code <association property="author" select="selectAuthor" column="author_id"/>}</li>
 *   <li>嵌套结果：{@code <association property="author" resultMap="authorMap"/>}</li>

/**

 *   <li>嵌套结果：{@code <collection property="posts" resultMap="postMap"/>}</li>

public record CollectionDef(

 * <resultMap id="vehicleMap" type="Vehicle">
```

### 5.1 类型安全 JOIN DSL（`cloud.opencode.base.db.join`）

> **解决 MyBatis-Plus 自 2016 年以来最大的痛点：不支持多表 JOIN。**

```java
/**
 * 类型安全的多表 JOIN 查询构造器。
 * <p>Type-safe multi-table JOIN query builder — the most requested feature
 * that MyBatis-Plus never delivered.</p>
 *
 * <pre>{@code
 * // 三表 JOIN + 条件 + 查询，全程类型安全
 * List<UserOrderVO> result = db.join(User.class)
 *     .leftJoin(Order.class, Order::getUserId, User::getId)
 *     .leftJoin(Product.class, Product::getId, Order::getProductId)
 *     .eq(User::getStatus, "active")
 *     .ge(Order::getAmount, 100)
 *     .select(User::getName, User::getEmail)
 *     .select(Order::getOrderNo, Order::getAmount)
 *     .select(Product::getProductName)
 *     .list(UserOrderVO.class);
 *
 * // 三表 JOIN + 分页
 * Page<UserOrderVO> page = db.join(User.class)
 *     .leftJoin(Order.class, Order::getUserId, User::getId)
 *     .leftJoin(Product.class, Product::getId, Order::getProductId)
 *     .eq(User::getStatus, "active")
 *     .select(User::getName, Order::getOrderNo)
 *     .page(Page.of(1, 10), UserOrderVO.class);
 *
 * // 子查询 IN
 * List<User> users = db.list(User.class, q -> q
 *     .inSql(User::getDeptId, db.subQuery(Dept.class)
 *         .select(Dept::getId)
 *         .eq(Dept::getStatus, "active")));
 *
 * // EXISTS 子查询
 * List<User> users = db.list(User.class, q -> q
 *     .exists(db.subQuery(Order.class)
 *         .eqColumn(Order::getUserId, User::getId)
 *         .ge(Order::getAmount, 1000)));
 * }</pre>
 */
public class JoinQuery<T> {
    public static <T> JoinQuery<T> of(Class<T> mainTable);

    // JOIN 声明（Lambda 列引用，编译期检查关联关系）
    public <J> JoinQuery<T> leftJoin(Class<J> table, SFunction<J, ?> on, SFunction<T, ?> eq);
    public <J> JoinQuery<T> rightJoin(Class<J> table, SFunction<J, ?> on, SFunction<T, ?> eq);
    public <J> JoinQuery<T> innerJoin(Class<J> table, SFunction<J, ?> on, SFunction<T, ?> eq);

    // 多条件 ON
    public <J> JoinQuery<T> leftJoin(Class<J> table, Consumer<JoinCondition<T, J>> onBuilder);

    // 条件（与 LambdaQuery 相同的方法签名）
    public <V> JoinQuery<T> eq(SFunction<?, V> column, V value);
    // ... ge/le/like/in/inSql/apply/last 等

    // 选择列（可从任何已 JOIN 的表中选择）
    @SafeVarargs
    public final JoinQuery<T> select(SFunction<?, ?>... columns);
    public JoinQuery<T> selectAll(Class<?> table);
    public JoinQuery<T> selectAs(SFunction<?, ?> column, String alias);

    // 分组/排序
    @SafeVarargs
    public final JoinQuery<T> groupBy(SFunction<?, ?>... columns);
    @SafeVarargs
    public final JoinQuery<T> orderByAsc(SFunction<?, ?>... columns);

    // 执行
    public <R> List<R> list(Class<R> resultType);
    public <R> Page<R> page(Page<R> page, Class<R> resultType);
    public long count();
    public String toSql();
    public Object[] getParams();
}

/**
 * 类型安全的子查询构造器。
 * <p>Type-safe subquery builder for IN/EXISTS clauses.</p>
 */
public class SubQuery<T> {
    public SubQuery<T> select(SFunction<T, ?> column);
    public <V> SubQuery<T> eq(SFunction<T, V> column, V value);
    public <V> SubQuery<T> eqColumn(SFunction<T, ?> col1, SFunction<?, ?> col2);  // 列与列比较
    // ... 所有条件方法
    public String toSql();
    public Object[] getParams();
}

/** 多条件 ON 子句构造器 — 用于复合 JOIN 条件 */
public class JoinCondition<L, R> {
    public JoinCondition<L, R> eq(SFunction<L, ?> left, SFunction<R, ?> right);    // ON l.col = r.col
    public JoinCondition<L, R> and(SFunction<L, ?> left, SFunction<R, ?> right);   // AND l.col = r.col
    public JoinCondition<L, R> andEq(SFunction<R, ?> column, Object value);        // AND r.col = ?
}
```

### 5.2 updateById 正确处理 null（修复 MyBatis-Plus 最大设计缺陷）

> **MyBatis-Plus 的 `updateById()` 默认跳过 null 字段，导致无法将字段更新为 null。这是中国开发者搜索最多的 MP 问题。**

```java
/**
 * 我们的 BaseMapper.updateById 默认行为：
 * - null 字段 → UPDATE SET field = NULL（正确的行为！）
 * - 如果用户确实想跳过 null，使用 updateByIdSelective()
 *
 * <pre>{@code
 * user.setOfflineTime(null);
 * mapper.updateById(user);              // ✅ SET offline_time = NULL（正确！）
 * mapper.updateByIdSelective(user);     // ✅ 跳过 null 字段（显式选择）
 * }</pre>
 */
public interface BaseMapper<T> {
    // ... 已有方法 ...

    /** 更新所有字段（null 字段也会被更新为 NULL） */
    int updateById(T entity);

    /** 选择性更新（跳过 null 字段，只更新非 null 字段） */
    int updateByIdSelective(T entity);
}
```

### 5.3 SQL 调试体验革命（`cloud.opencode.base.db.debug`）

> **开发者最常见的调试动作：从日志中复制 SQL，手动替换 ? 为参数值，粘贴到数据库客户端执行。我们让这个流程自动化。**

```java
/**
 * SQL 调试工具 — 开发阶段的终极调试体验。
 * <p>Revolutionary SQL debugging experience for developers.</p>
 *
 * <pre>{@code
 * // 开启调试模式
 * OpenDb db = OpenDb.of(dataSource).debug(true);
 *
 * // 日志输出（自动内联参数，可直接复制执行）：
 * // ┌─────────────────────────────────────────────────────
 * // │ SQL: SELECT * FROM users WHERE status = 'active' AND age >= 18
 * // │ Source: UserService.java:42 → UserMapper.selectList()
 * // │ Time: 3.2ms | Rows: 15
 * // └─────────────────────────────────────────────────────
 *
 * // 也支持编程式调试

    public static ExplainResult explain(JdbcTemplate template, String sql, Object[] params);

    /** 获取调用源码位置（通过 StackWalker） */
    public static String getSourceLocation();
}

/** EXPLAIN 结果 — 仅保留原始输出，不做结构化解析（各数据库 EXPLAIN 格式差异极大） */
public record ExplainResult(
    String rawOutput                   // 原始 EXPLAIN 输出（MySQL/PostgreSQL/Oracle 格式各异）
) { }

/**
 * SQL 注释注入 — 自动在 SQL 中追加源码位置注释，便于 DBA 追踪慢查询来源。
 * <p>Auto-inject source location as SQL comment for DBA troubleshooting.</p>
 *
 * <pre>{@code
 * // 开启后，所有 SQL 自动追加注释：
 * // SELECT * FROM users WHERE status = ?
 * //   /* UserService.findActive:42 | trace_id=abc123 */
 * db.addInterceptor(SqlCommentInterceptor.builder()

 *     .applicationName("order-service")
 *     .build());
 * }</pre>
 */
public class SqlCommentInterceptor implements DbInterceptor {
    public static Builder builder();

    public static class Builder {
        public Builder sourceLocation(boolean enabled);     // 追加 class:line
        public Builder traceId(Supplier<String> supplier);  // 追加链路追踪 ID
        public Builder applicationName(String name);        // 追加应用名
        public Builder custom(String key, Supplier<String> value);  // 自定义字段
        public SqlCommentInterceptor build();
    }
}
```

### 5.4 方法名查询派生（`cloud.opencode.base.db.mapper.derived`）

> **借鉴 Spring Data / Micronaut Data，从方法名自动推导 SQL，无需写任何查询。**

```java
/**
 * 方法名查询 — 从方法签名自动推导 SQL 查询。
 * <p>Derive SQL queries from method names, Spring Data style.</p>
 *
 * <pre>{@code
 * public interface UserMapper extends BaseMapper<User> {
 *
 *     // 方法名自动推导
 *     List<User> findByStatus(String status);
 *     // → SELECT * FROM users WHERE status = ?
 *
 *     List<User> findByNameLikeAndAgeGreaterThan(String name, int age);
 *     // → SELECT * FROM users WHERE name LIKE ? AND age > ?
 *
 *     Optional<User> findByEmail(String email);
 *     // → SELECT * FROM users WHERE email = ? (LIMIT 1)
 *
 *     long countByStatus(String status);
 *     // → SELECT COUNT(*) FROM users WHERE status = ?
 *
 *     boolean existsByEmail(String email);
 *     // → SELECT COUNT(1) FROM users WHERE email = ? LIMIT 1
 *
 *     List<User> findByDeptIdOrderByCreatedAtDesc(Long deptId);
 *     // → SELECT * FROM users WHERE dept_id = ? ORDER BY created_at DESC
 *
 *     int deleteByStatusAndCreatedAtBefore(String status, LocalDateTime before);
 *     // → DELETE FROM users WHERE status = ? AND created_at < ?
 *
 *     Page<User> findByStatus(String status, Page<User> page);
 *     // → 自动分页 + COUNT
 * }
 * }</pre>
 *
 * <p>支持的关键词：</p>
 * <ul>
 *   <li>查询：{@code findBy} / {@code readBy} / {@code queryBy} / {@code getBy}</li>
 *   <li>计数：{@code countBy}</li>
 *   <li>存在：{@code existsBy}</li>
 *   <li>删除：{@code deleteBy} / {@code removeBy}</li>
 *   <li>条件：{@code And} / {@code Or}</li>
 *   <li>比较：{@code GreaterThan} / {@code LessThan} / {@code GreaterThanEqual} / {@code LessThanEqual}</li>
 *   <li>模糊：{@code Like} / {@code NotLike} / {@code StartingWith} / {@code EndingWith} / {@code Containing}</li>
 *   <li>范围：{@code Between} / {@code In} / {@code NotIn}</li>
 *   <li>NULL：{@code IsNull} / {@code IsNotNull}</li>
 *   <li>排序：{@code OrderByXxxAsc} / {@code OrderByXxxDesc}</li>
 *   <li>去重：{@code Distinct}</li>
 *   <li>限制：{@code First} / {@code Top} / {@code TopN}（如 {@code findFirst3By...}）</li>
 * </ul>
 */
public class DerivedQueryParser {
    /** 从方法名解析出 SQL 语句 */
    public static ParsedQuery parse(Method method, EntityMeta meta);
}

public record ParsedQuery(String sql, String countSql, List<String> paramNames,
                           boolean isPaged, boolean isOptional) { }
```

### 5.5 游标分页 / Keyset 分页（`cloud.opencode.base.db.page`）

> **OFFSET 分页在大数据量下性能极差（OFFSET 10000 需扫描前 10000 行再丢弃）。游标分页是正确的做法。**

```java
/**
 * 游标分页（Keyset Pagination）— 大数据量下性能恒定的分页方案。
 * <p>Cursor-based pagination with constant performance regardless of page depth.</p>
 *
 * <pre>{@code
 * // 传统 OFFSET 分页（第 1000 页时极慢）
 * Page<User> page = db.page(User.class, Page.of(1000, 10), q -> q.orderByDesc(User::getId));
 * // → SELECT * FROM users ORDER BY id DESC LIMIT 10 OFFSET 9990  ❌ 慢
 *
 * // 游标分页（任何页都一样快）
 * CursorPage<User> page = db.cursorPage(User.class, CursorPage.of(10), q -> q
 *     .eq(User::getStatus, "active")
 *     .orderByDesc(User::getId));
 * // 首页: SELECT * FROM users WHERE status='active' ORDER BY id DESC LIMIT 11
 * // 翻页: SELECT * FROM users WHERE status='active' AND id < 12345 ORDER BY id DESC LIMIT 11
 *
 * String nextCursor = page.getNextCursor();    // 传给前端，前端下次请求带回
 * List<User> records = page.getRecords();
 * boolean hasNext = page.hasNext();
 * }</pre>
 */
public class CursorPage<T> {
    public static <T> CursorPage<T> of(long size);
    public static <T> CursorPage<T> of(long size, String cursor);    // 从 cursor 继续

    public List<T> getRecords();
    public String getNextCursor();          // Base64 编码的游标值
    public String getPreviousCursor();      // 反向游标
    public boolean hasNext();
    public boolean hasPrevious();
    public long getSize();
}
```

```java
/**
```

### 5.8 乐观锁 + 自动重试

> **乐观锁冲突后开发者总是忘记处理。我们内置自动重试机制。**

```java
/**
 * 乐观锁自动重试 — 冲突时自动重新加载并重试。
 * <p>Automatic retry on optimistic lock conflict.</p>
 *
 * <pre>{@code
 * // 传统方式：需手动处理冲突
 * int rows = mapper.updateById(order);  // rows=0 表示版本冲突
 * if (rows == 0) {
 *     // 手动重新加载、重新修改、重新提交... 😮‍💨
 * }
 *
 * // 我们的方式：自动重试
 * db.optimistic(Order.class, 1L, order -> {
 *     order.setStatus("completed");
 *     order.setAmount(order.getAmount().add(bonus));
 * });
 * // 内部：加载 → 修改 → updateById → 若冲突自动重试（最多 3 次）
 * }</pre>
 */
// OpenDb 新增方法
public class OpenDb {
    /** 乐观锁安全更新：自动加载 → 修改 → 保存 → 冲突重试 */
    public <T> T optimistic(Class<T> type, Serializable id, Consumer<T> modifier);
    public <T> T optimistic(Class<T> type, Serializable id, Consumer<T> modifier, int maxRetries);
}
```

### 6.1 API 对比

#### 条件查询

| 启动速度 | 需要解析大量 XML、初始化 | 极致轻量，毫秒级启动 |

### 6.3 MyBatis-Flex 对比与借鉴

MyBatis-Flex 是近年来最成功的国产 ORM 框架之一（GitHub 2000+ Star），定位与我们高度重叠。必须正面分析其设计优劣，明确差异化。

#### 6.3.1 MyBatis-Flex 概述

```
架构定位：MyBatis 增强层（仍依赖 mybatis-core）
核心特色：APT 编译期代码生成 + 高性能 QueryWrapper
许可协议：Apache 2.0
Java 版本：Java 8+
```

**核心设计**：通过 APT（Annotation Processing Tool）在编译期为每个实体生成 `TableDef` 类，
包含 `QueryColumn` 常量。查询时直接引用常量（`ACCOUNT.ID`），避免反射和 Lambda 序列化。

```java
// MyBatis-Flex 的 APT 生成类（编译期自动生成）
public class Tables {
    public static final AccountTableDef ACCOUNT = new AccountTableDef();
}

// 使用生成的常量构建查询
QueryWrapper query = QueryWrapper.create()
    .select(ACCOUNT.ID, ACCOUNT.USER_NAME)
    .where(ACCOUNT.AGE.ge(18))
    .orderBy(ACCOUNT.ID.desc());
```

#### 6.3.2 功能对比

| 特性 | MyBatis-Flex | opencode-base-db |
|------|-------------|------------------|
| **底层依赖** | mybatis-core（必须） | 零外部依赖，自建引擎 |
| **最低 Java 版本** | Java 8 | Java 25（预览特性） |
| **查询构建方式** | APT 编译期 `QueryColumn` 常量 | Lambda `SFunction` 提取列名 |
| **多表 JOIN** | `leftJoin(ACCOUNT).on(...)` QueryColumn 语法 | `leftJoin(Order.class, User::getId, Order::getUserId)` Lambda 语法 |
| **关联查询** | 3 种：Relations 注解 / FieldQuery / JoinQuery | JoinQuery DSL + 嵌套结构映射 |
| **Db + Row 模式** | 内置（无实体 schema-less 操作） | `Db` 工具类（Lambda 类型安全） |
| **NULL 值查询跳过** | 条件方法自动跳过 null 参数 | 需显式 `.eqIfNotNull()` 或等效方法 |
| **UNION / UNION ALL** | 原生支持 | Phase 3 JoinQuery 扩展支持 |
| **Wrapper 序列化** | 支持 RPC 传输（可序列化） | 暂不支持 |
| **复合主键** | `@Id` 多字段复合主键 | `@Id` 支持复合主键 |
| **数据脱敏** | 免费内置 | 免费内置（`@Sensitive`） |
| **数据加密** | 免费内置 | 免费内置（`FieldEncryptInterceptor`） |
| **多租户** | 内置 | 内置（`TenantInterceptor`） |
| **乐观锁** | `@Version` | `@Version` + 自动重试 |
| **逻辑删除** | `@Column(isLogicDelete=true)` | `@LogicDelete` |
| **分页** | 内置分页插件 | `Page` + `CursorPage` 游标分页 |
| **SQL 审计** | 内置 AuditManager | `SqlAuditInterceptor` |
| **代码生成** | 内置 | 延后支持 |
| **Spring Boot Starter** | 有 | 计划补充 |

#### 6.3.3 API 风格对比

**基础查询**

```java
// MyBatis-Flex — APT 常量引用
QueryWrapper query = QueryWrapper.create()
    .where(ACCOUNT.AGE.ge(18))
    .and(ACCOUNT.STATUS.eq("active"));
List<Account> list = accountMapper.selectListByQuery(query);

// opencode-base-db — Lambda 类型安全
List<Account> list = db.list(Account.class, q -> q
    .ge(Account::getAge, 18)
    .eq(Account::getStatus, "active"));
```

**多表 JOIN**

```java
// MyBatis-Flex — QueryColumn JOIN
QueryWrapper query = QueryWrapper.create()
    .select(ARTICLE.ALL_COLUMNS, ACCOUNT.USER_NAME)
    .from(ARTICLE)
    .leftJoin(ACCOUNT).on(ARTICLE.ACCOUNT_ID.eq(ACCOUNT.ID))
    .where(ACCOUNT.AGE.ge(18));

// opencode-base-db — Lambda JOIN
List<ArticleVO> list = db.join(Article.class)
    .leftJoin(Account.class, Article::getAccountId, Account::getId)
    .ge(Account::getAge, 18)
    .list(ArticleVO.class);
```

**Db 工具**

```java
// MyBatis-Flex — Db + Row
List<Row> rows = Db.selectListByQuery("tb_account",
    QueryWrapper.create().where(ACCOUNT.AGE.ge(18)));
Row row = rows.get(0);
String name = row.getString("user_name");

// opencode-base-db — Db Lambda
List<Account> list = Db.list(Account.class, q -> q.ge(Account::getAge, 18));
```

#### 6.3.4 MyBatis-Flex 的优势（值得借鉴）

| 优势 | 分析 | 我们的应对 |
|------|------|-----------|
| **APT 编译期生成** | 零反射开销，IDE 自动补全，编译期发现列名错误 | 我们使用 Lambda 序列化提取列名，运行期有微小开销但无需额外编译步骤；可考虑未来补充 APT 可选模块 |
| **Java 8+ 兼容** | 覆盖 99% 存量项目 | 我们定位 Java 25+，放弃存量但拥抱未来（见 11.1 讨论） |
| **Null 参数自动跳过** | `.where(ACCOUNT.AGE.ge(ageParam))` 当 ageParam 为 null 时自动忽略该条件，减少 if 判断 | **借鉴**：在 LambdaQuery 条件方法中增加 null 跳过行为（或提供 `eqIfPresent` / `geIfPresent` 系列方法） |
| **QueryWrapper 可序列化** | 支持 RPC 场景传输查询条件 | 评估是否需要支持 |
| **UNION / UNION ALL** | 原生链式 API | **借鉴**：在 JoinQuery 中增加 `.union()` / `.unionAll()` 支持 |
| **Relations 注解** | `@RelationOneToMany` 声明式关联加载 | 我们使用 JoinQuery DSL 显式查询，避免 N+1 问题 |
| **Row 模式** | 无需定义实体即可操作数据 | `Db` 工具类已支持类似能力 |

#### 6.3.5 MyBatis-Flex 的局限（我们的机会）

| 局限 | 我们的优势 |
|------|-----------|
| **仍依赖 MyBatis 核心** | 我们是独立引擎，零外部依赖，不继承 MyBatis 的设计债务 |
| **无法增强 XML SQL** | XML 中的 SQL 仍是原始 MyBatis 处理，Flex 的增强仅限 Java API |
| **无 JPMS 模块化** | 我们的 `module-info.java` 完整声明，强封装 |
| **无 Record 映射** | 我们原生支持 `record` 类型结果映射 |
| **无 sealed class** | 我们的 `sealed interface SqlNode` 编译期保证节点类型完备 |
| **无虚拟线程优化** | 我们的连接池、异步查询原生适配虚拟线程 |
| **APT 增加构建复杂度** | 必须配置 annotation processor，IDE 支持偶有问题；我们的 Lambda 方式开箱即用 |
| **SQL 调试能力弱** | 我们的 `SqlDebugger` 提供参数内联 + 源码定位 + EXPLAIN 一体化调试体验 |
| **游标分页缺失** | 仅支持 OFFSET 分页，大数据量性能差；我们提供 `CursorPage` keyset 分页 |
| **乐观锁无自动重试** | 冲突后需手动处理；我们的 `db.optimistic()` 支持自动重试 |

#### 6.3.6 借鉴结论

从 MyBatis-Flex 的成功经验中，我们应采纳以下改进：

1. **Null 安全条件方法**：为 LambdaQuery 所有条件方法提供 `xxxIfPresent` 变体（如 `eqIfPresent`、`geIfPresent`），当值为 null 时自动跳过该条件
2. **UNION 支持**：在 JoinQuery DSL 中补充 `.union(subQuery)` 和 `.unionAll(subQuery)` 链式方法
3. **Spring Boot Starter 优先级提升**：MyBatis-Flex 的成功证明 Starter 是用户采用的关键门槛，应在首个版本即提供
4. **可选 APT 模块**（远期）：提供 `opencode-base-db-apt` 可选模块，为追求极致性能的用户生成编译期 `TableDef` 常量

**不采纳的设计**：
- **Row 模式作为核心 API**：schema-less 操作牺牲类型安全，我们更倾向 Lambda 类型安全查询
- **Relations 注解式关联加载**：隐式加载容易导致 N+1 问题，我们坚持 JoinQuery 显式查询

## 8. 风险与缓解

| 风险 | 级别 | 缓解措施 |
|------|------|----------|
| OGNL 表达式边界场景 | 中 | `OgnlBridge` 预处理器覆盖全部 OGNL 语法（含 `@Class@staticMethod`、投影、`#root`/`#this`），配套完整测试用例集 |
| 嵌套查询 N+1 性能 | 中 | association/collection 嵌套查询默认 lazy loading，文档引导用户优先使用嵌套结果（一次查询） |
| JoinQuery 在复杂场景下的 SQL 正确性 | 中 | 多表 JOIN + 子查询 + 分页组合场景复杂，通过大量集成测试覆盖边界情况 |
| Lambda 列名提取在某些 JDK 上有限制 | 低 | `SerializedLambdaWrapper` 已在 JDK 25 上验证通过 |
| API 与 MyBatis-Plus 不完全一致 | 低 | 这是设计意图，我们提供迁移指南和名称映射表 |
| 存储过程 CALLABLE 兼容性 | 低 | 依赖 JDBC 驱动实现，我们只做 Statement 包装和 OUT 参数映射 |
| OpenTelemetry 可选依赖引入 | 低 | `opentelemetry-api` 设为 `<optional>true</optional>`，运行时无此 JAR 则拦截器自动禁用（`ClassNotFoundException` 检测） |

## 11. 慎重考虑：开源视角下的战略审视

> 以下内容从开源项目成功的角度进行冷静分析，指出方案在技术之外需要面对的现实问题。

### 11.1 Java 版本要求过高

| 框架 | 最低 Java 版本 |
|------|---------------|
| MyBatis | 8 |
| MyBatis-Plus | 8 |
| MyBatis-Flex | 8 |
| jOOQ 开源版 | 8 |
| Spring Boot 3 | 17 |
| Hibernate 6 | 11 |
| **opencode-base-db** | **25** |

当前中国 95%+ 的 Java 生产环境运行在 8/11/17/21 上。要求 Java 25 意味着：
- 绝大多数公司无法在生产中使用
- 开发者无法在现有项目中试用
- 开源社区难以参与贡献（IDE/CI 环境尚未普及 25）

**可选缓解**：最低要求降为 Java 21（当前 LTS），Java 25 特性（sealed class、Record pattern 等）做运行时检测可选增强。或者明确定位为"面向未来"的先锋框架，接受初期用户量较少。

### 11.2 最大竞品：MyBatis-Flex

**MyBatis-Flex** 是定位高度相似且已获得成功的国产 ORM 框架（GitHub 2000+ Star）。已在 **6.3 节** 中补充了详细的功能对比、API 风格对比、借鉴结论。

核心结论：
- MyBatis-Flex 的 **APT 编译期代码生成** 和 **null 参数自动跳过** 值得借鉴
- 我们的差异化在于：零外部依赖、JPMS 模块化、Record/sealed class、虚拟线程、SqlDebugger、CursorPage
- 如果开源发布，需要能清晰回答"和 MyBatis-Flex 比有什么优势？"这个问题（6.3.5 已列出 10 项具体优势）

### 11.3 缺少 Spring Boot Starter

方案明确写了"Spring 集成注解 (`@MapperScan`) 属于 Spring 生态，不在本框架范围"。技术上正确，但在开源策略上有重大影响。

中国 Java 生态现实：**Spring Boot + MyBatis-Plus** 是事实标准。没有 `spring-boot-starter-opencode-db`：
- 无法在 Spring Boot 项目中开箱即用（99% 的企业项目）
- 无法自动配置 DataSource
- 无法集成 Spring 事务管理（`@Transactional`）
- 无法被 Spring 生态的其他组件发现和集成

MyBatis-Plus 的成功很大程度上归功于 `mybatis-plus-boot-starter` 做到了"加个依赖就能用"。

**建议**：在 Phase 1 完成后，优先补充 `opencode-base-db-spring-boot-starter` 模块（自动配置 DataSource、事务管理器注入、Mapper 扫描），降低用户接入成本。

### 11.4 初始发布范围过大

成功开源项目的普遍规律：**小而精发布，持续迭代**。

| 项目 | 初始版本 |
|------|---------|
| Vue.js | 一个 MVVM 数据绑定库 |
| Spring Boot | 一个约定优于配置的启动器 |
| MyBatis-Plus | MyBatis 的 BaseMapper 增强 |

当前方案 173 个文件的 v1.0 意味着：
- 开发周期长，发布前无法获得用户反馈
- 大量功能没经过实战验证
- Bug 面铺得太大，质量保障困难
- 新贡献者面对巨大代码库望而却步

**建议分阶段发布**：

| 版本 | 内容 | 文件数 | 目标 |
|------|------|-------|------|
| v0.1 | Phase 1（BaseMapper + Wrapper + 拦截器 + IService） | ~41 | 尽快获得用户反馈，验证 API 设计 |
| v0.5 | + Phase 3 领先特性（JoinQuery、SqlDebugger、CursorPage） | ~52 | 建立差异化竞争力 |
| v1.0 | + Phase 2 XML Mapper 兼容层 | ~173 | 完整迁移能力，吸引旧项目用户 |

### 11.5 "100% XML 兼容"承诺的风险

方案反复强调"mapper.xml 零修改迁移"、"100% 语法兼容"。但需注意：

- MyBatis 有 **15 年** 的边缘场景积累
- OGNL 表达式在实际项目中用法极多变
- TypeHandler、ResultMap 嵌套、discriminator 的组合场景极其复杂
- 方案预估 ~480 个测试 vs MyBatis 自身 **数千个** 测试

一旦用户拿旧项目迁移时发现某个 XML 不兼容，对项目的信任会受到严重损害。

**建议**：将措辞从"100% 兼容"调整为"兼容 MyBatis 3.5.x 核心语法和常用功能"，同时建立一个**兼容性测试矩阵**（从 MyBatis 官方测试用例中提取），用事实说话而非承诺。

### 11.6 项目定位的表述

> "我们不是 MyBatis 的复刻品，而是它的继承者。"

开源社区的潜规则：**用作品说话，不自封王位。** 在代码发布前自称"继承者"，容易被社区视为傲慢。对比成功案例：
- Redis 从未攻击 Memcached，只是做了一个更好的东西
- Go 从未自称"C 的继承者"
- Vue 从未自称"Angular/React 的继承者"

**建议**：改为更务实的表述，如"面向 Java 21+ 的现代 ORM 框架，借鉴 MyBatis 生态的优秀设计"。让社区自己得出"这比 MyBatis 好"的结论。

### 11.7 三种可选战略方向

#### 方向 A：独立现代 ORM（推荐度：★★★★★）

砍掉 Phase 2（XML 兼容层 ~121 文件），只做 Phase 1 + Phase 3：

```
opencode-base-db
├── BaseMapper<T> + CRUD
├── LambdaQuery / LambdaUpdate（类型安全）
├── Page 分页 + CursorPage 游标分页
├── 14 个拦截器
├── JoinQuery DSL（核心卖点）
├── SqlDebugger（核心卖点）
├── IService 服务层
└── Db 静态工具
```

- ~52 个文件，可在合理时间内高质量交付
- 不承诺 XML 兼容，没有兑现不了的承诺
- 卖点清晰："零 XML、零依赖、Java 21+ 现代 ORM"
- 目标用户：新项目、追求技术前沿的团队

#### 方向 B：全量计划，分阶段开源（推荐度：★★★★）

保持全部计划，但按 v0.1 → v0.5 → v1.0 分阶段发布（见 11.4 表格），每个版本独立可用、独立有价值。Spring Boot Starter 从 v0.1 就提供。

#### 方向 C：内部生态组件（推荐度：★★★）

如果主要服务于 opencode-base 体系的 43 个模块内部使用，而非对外开源竞争，那当前方案完全可行。但文档的对比措辞应调整，不必对标 MyBatis/Plus 的公开竞品定位。

### 11.8 总结

| 维度 | 评分 | 说明 |
|------|------|------|
| 技术设计 | ★★★★★ | API 优雅、架构清晰、充分利用现代 Java |
| 差异化能力 | ★★★★ | JoinQuery、SqlDebugger、零依赖、虚拟线程有真正竞争力 |
| 目标用户定义 | ★★ | 未明确是新项目还是迁移项目，未区分内部使用还是开源竞争 |
| 竞品分析 | ★★★★ | 已补充 MyBatis-Flex 对比与借鉴（6.3），差异化清晰 |
| 发布策略 | ★★ | 173 文件一次性发布风险高，缺少 MVP 思维 |
| 生态集成 | ★ | 无 Spring Boot Starter，Java 25 门槛过高 |

**核心观点**：好的开源项目不是功能最多的，而是**能让程序员在 5 分钟内跑起来并说"真香"的**。建议优先保证 Phase 1 + Phase 3 的极致质量和开箱即用体验，再考虑 Phase 2 的兼容层扩展。
