# OpenCode 组件拆分规划

> opencode-base 瘦身为纯基础工具库，将可独立发展的领域组件抽离为独立项目。

## 1. 拆分原则

### 1.1 判断标准

一个模块应该从 opencode-base 独立出去，当且仅当它满足以下条件中的 **3 项以上**：

| 序号 | 条件 | 说明 |
|------|------|------|
| 1 | **领域深度** | 该领域有大量可扩展方向（插件、策略、后端适配、协议支持等） |
| 2 | **文件规模** | 当前已超过 50 个源文件，或扩展后预计超过 100 个 |
| 3 | **独立用户价值** | 脱离 opencode-base 体系，单独使用仍有明确价值 |
| 4 | **竞品对标** | 存在成熟的独立开源竞品（说明该领域足以支撑一个项目） |
| 5 | **生态潜力** | 可发展出自己的插件体系、配置体系、文档站 |
| 6 | **发布节奏差异** | 该模块的迭代速度与基础工具类有本质差异 |

### 1.2 保留标准

留在 opencode-base 的模块应符合：

- API 稳定，变动频率低
- 功能聚焦，不太会膨胀
- 属于「用完即走」的工具类
- 无需独立的插件/扩展体系

### 1.3 依赖隔离原则（铁律）

> **基础模块不许依赖独立项目。依赖方向只能是：独立项目 → opencode-base，不可反向。**

这意味着：如果某个模块被多个基础模块依赖，它**不能完全抽走**。解决方案是 **「基础版留守 + 独立项目继承扩展」**：

```
OpenCache（独立项目，丰富功能）
    │  extends / depends on
    ▼
opencode-base-cache（基础模块，保留在 base 中）
    ▲  optional dependency
    │
opencode-base-feature, opencode-base-config, ...（其他基础模块）
```

**适用此模式的模块**：

| 基础模块 | 保留在 base 的内容 | 独立项目扩展的内容 |
|---------|------------------|------------------|
| `opencode-base-cache` | 本地缓存核心（LRU/LFU/TTL/事件/指标） | Redis/Memcached 适配、Spring Cache 桥接、分布式同步、管理控制台 |
| `opencode-base-crypto` | 加密算法核心（AES/RSA/签名/哈希/JWT） | HSM 集成、Vault 适配、证书自动管理、安全审计、密码策略 |

**不适用此模式的模块**（无基础模块依赖它们，可完全抽走）：

| 模块 | 基础模块依赖数 | 处理方式 |
|------|--------------|---------|
| db | 0 | 完全抽走 |
| resilience | 0 | 完全抽走 |
| validation | 1（config 可选） | 完全抽走，config 移除可选依赖 |
| security | 0 | 合并入 OpenSecurity，依赖 base-crypto |
| net | 0 | 完全抽走 |
| observability | 1（pool 可选） | 完全抽走，pool 移除可选依赖 |
| tasker | 0 | 完全抽走 |

### 1.4 目录与项目结构规则

所有独立项目统一存放在 `opencode-pro` 目录下(完整地址 /Users/jon/dev/java/opencode-pro )，每个独立项目拥有**独立的 Git 仓库和独立的 IDEA 项目**（不像 opencode-base 那样所有模块共享同一个 IDEA 项目）。

**磁盘目录结构**：

```
/Users/jon/dev/java/
├── opencode-base/                  ← 基础工具库（1 个 IDEA 项目，36 个模块）
│   ├── opencode-base-core/
│   ├── opencode-base-cache/
│   ├── opencode-base-crypto/
│   ├── ... (其他 33 个模块)
│   └── pom.xml                     ← 统一父 POM
│
└── opencode-pro/                   ← 独立项目集合（非 IDEA 项目，纯目录）
    ├── OpenData/                   ← 独立 IDEA 项目 + 独立 Git 仓库
    │   ├── opencode-opendata-core/
    │   ├── opencode-opendata-mapper/
    │   ├── opencode-opendata-spring-boot-starter/
    │   ├── pom.xml
    │   └── .git/
    ├── OpenCache/                  ← 独立 IDEA 项目 + 独立 Git 仓库
    │   ├── opencode-opencache-core/
    │   ├── opencode-opencache-redis/
    │   ├── pom.xml
    │   └── .git/
    ├── OpenRes/                    ← 独立 IDEA 项目 + 独立 Git 仓库
    ├── OpenValidation/
    ├── OpenSecurity/
    ├── OpenNet/
    ├── OpenMetrics/
    └── OpenTasker/
```

**关键规则**：

| 规则 | 说明 |
|------|------|
| **独立 Git 仓库** | 每个 Open* 项目有自己的 `.git/`，独立管理版本、分支、Tag |
| **独立 IDEA 项目** | 每个 Open* 目录可直接用 IDEA 打开，互不干扰 |
| **独立 CI/CD** | 每个项目有自己的构建管线，独立发布 |
| **opencode-pro 只是目录** | `opencode-pro/` 本身不是 Git 仓库、不是 IDEA 项目，仅用于组织磁盘结构 |
| **opencode-base 不变** | opencode-base 仍是单个 IDEA 项目 + 单个 Git 仓库，所有基础模块在其中 |

|------|--------|---------|------|

## 5. 依赖关系处理

### 5.1 铁律：依赖方向只能向下

```
独立项目 ──→ opencode-base    ✅ 允许
opencode-base ──→ 独立项目    ❌ 禁止
独立项目 ──→ 独立项目          ⚠️ 仅 optional/SPI
```

### 5.2 独立项目对 opencode-base 的依赖

独立项目可以依赖 base 中的任意模块（方向正确）：

| 独立项目 | 依赖的 base 模块 | scope |
|---------|----------------|-------|
| OpenData | base-core, base-reflect, base-expression, base-xml | compile |
| OpenCache | base-core, **base-cache** | compile |
| OpenResilience | base-core | compile |
| OpenValidation | base-core | compile |
| OpenSecurity | base-core, **base-crypto** | compile |
| OpenNet | base-core | compile |
| OpenMetrics | base-core | compile |
| OpenTasker | base-core | compile |

### 5.3 独立项目之间的依赖

原则：独立项目之间 **不直接依赖**，通过 SPI 或 `optional` 依赖解耦。

```
OpenData       ──(optional)──> OpenCache       # 二级缓存
OpenData       ──(optional)──> OpenRes   # 连接重试
OpenNet        ──(optional)──> OpenRes   # HTTP 重试/熔断
OpenTasker     ──(optional)──> OpenMetrics      # 任务指标
```

### 5.4 opencode-base 内部简化

独立项目抽出后，opencode-base 的依赖图变得更简洁：

```
核心层:    core (零依赖)
基础层:    string, reflect, collections, io, date, hash, deepclone,
          classloader, i18n, id, expression, lock, rules, crypto,
          cache (仅依赖 core 或基础层)
领域层:    config, functional, json, log, parallel, pool, serialization,
          xml, yml, oauth2 (依赖 core + 少量基础层)
业务层:    event, web, money, lunar, captcha, email, sms, feature,
          geo, image, tree, graph, timeseries, pdf, test
```

## 8. 迁移步骤

### 8.1 Phase 0：准备（不改代码）

1. 创建 `/Users/jon/dev/java/opencode-pro/` 目录
2. 在 `opencode-pro/` 下为每个独立项目初始化独立 Git 仓库（`git init OpenData` 等）
3. 确认 CI/CD 模板（统一构建配置）
4. 确定版本号策略：所有独立项目从 `1.0.0` 开始，opencode-base 保持当前版本
5. 将 opencode-base 先 `mvn install` 到本地仓库，供独立项目依赖

### 8.2 Phase 1：完全抽走的模块迁移

以 OpenData 为例（完全抽走模式）：

```
1. cd /Users/jon/dev/java/opencode-pro/
2. mkdir OpenData && cd OpenData && git init
3. 创建 opencode-opendata-core 模块，设置 pom.xml
4. 复制 opencode-base-db/src → opencode-opendata-core/src
5. 批量替换包名：cloud.opencode.base.db → cloud.opencode.opendata
6. 更新 module-info.java
7. 在 pom.xml 中声明对 opencode-base-core 等的依赖
8. IDEA 中打开 opencode-pro/OpenData/ 作为独立项目
9. 确保编译通过 + 全部测试通过
10. 从 opencode-base 的父 POM 中移除 opencode-base-db 模块
11. 在 opencode-base 中删除 opencode-base-db 目录
```

适用于：OpenData、OpenRes、OpenValidation、OpenNet、OpenMetrics、OpenTasker（6 个）

### 8.3 Phase 2：继承扩展的模块创建

以 OpenCache 为例（继承扩展模式）：

```
1. cd /Users/jon/dev/java/opencode-pro/
2. mkdir OpenCache && cd OpenCache && git init
3. 创建 opencode-opencache-core 模块，设置 pom.xml
4. 在 pom.xml 中声明对 opencode-base-cache 的 compile 依赖
5. 编写扩展代码（Redis 适配、Spring Cache 桥接等），继承 base-cache 的类/接口
6. opencode-base-cache 不做任何修改，不删除任何文件
7. IDEA 中打开 opencode-pro/OpenCache/ 作为独立项目
8. 确保编译通过 + 全部测试通过
```

适用于：OpenCache（继承 base-cache）、OpenSecurity（继承 base-crypto + 抽走 security）

### 8.4 Phase 3：交叉依赖处理（铁律：base 不可依赖独立项目）

1. opencode-base 中依赖已独立模块的其他模块（如 `pool` 依赖 `observability`），**移除该依赖**（不是改为 optional，而是彻底移除）
2. 如果确实需要功能，通过 SPI 在独立项目侧提供实现
3. 独立项目之间的依赖（如 `OpenData` 使用 `OpenCache`），声明为 `optional`
4. 运行全量测试，确保无破坏

### 8.5 Phase 4：功能扩展

每个独立项目按照各自路线图独立迭代，不再受 opencode-base 的发布节奏约束。

### 8.6 迁移顺序

按依赖关系从底层到上层迁移，减少交叉影响：

| 顺序 | 项目 | 模式 | 原因 |
|------|------|------|------|
| 1 | OpenSecurity | 继承扩展 + 抽走 | crypto 留守；security 无被依赖，直接抽走 |
| 2 | OpenMetrics | 完全抽走 | pool 移除可选依赖后即可抽走 |
| 3 | OpenValidation | 完全抽走 | config 移除可选依赖后即可抽走 |
| 4 | OpenNet | 完全抽走 | 无被依赖 |
| 5 | OpenRes | 完全抽走 | 无被依赖 |
| 6 | OpenCache | 继承扩展 | base-cache 不动，新建扩展项目 |
| 7 | OpenTasker | 完全抽走 | 无被依赖 |
| 8 | OpenData | 完全抽走 | 最复杂，最后迁移 |

## 12. 决策总结

| 维度 | 说明 |
|------|------|
| **完全抽走** | 7 个模块 → 6 个独立项目（db, resilience, validation, security, net, observability, tasker） |
| **继承扩展** | 2 个独立项目继承 base 基础版（OpenCache ← base-cache, OpenSecurity ← base-crypto） |
| **保留在 base** | 36 个模块（含 cache、crypto 基础版 + 纯工具、数据格式、轻量业务） |
| **铁律** | 基础模块不许依赖独立项目，依赖方向只能：独立项目 → opencode-base |
| **核心原则** | opencode-base = 「用完即走的工具箱」；独立项目 = 「可发展生态的框架」 |
| **迁移策略** | 按依赖从底到上逐个迁移，先迁无被依赖的模块 |
| **版本管理** | 各项目独立版本 + 统一 BOM |
| **命名规则** | 独立项目统一 `opencode-Open*` 命名；包名 `cloud.opencode.open*.*` |
